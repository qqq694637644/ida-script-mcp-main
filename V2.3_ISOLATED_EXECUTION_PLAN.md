# V2.3 Isolated Execution + GUI Change Replay Implementation Plan

This document is written for another AI/code maintainer to review and implement. The expected shell is **PowerShell 7 (`pwsh`)** from the repository root.

## 0. Current branch context

Current branch already contains a V1.5-style refactor:

- `src/ida_script_mcp/protocol.py`
  - strict `ExecuteRequest` / `ExecuteResult` / `ExecutionError` models
  - statuses: `ok`, `timeout`, `script_error`, `source_error`, `busy`, `plugin_response_timeout`
- `src/ida_script_mcp/execution.py`
  - in-process `ScriptExecutor`
  - Python bytecode soft timeout via trace hook
- `src/ida_script_mcp/ida_plugin.py`
  - `/execute` endpoint uses strict request validation and `ExecutionManager`
  - `/health` exposes execution state
- `src/ida_script_mcp/server.py`
  - MCP-side execute request forwards `timeout_seconds`
  - distinguishes Python timeout from plugin response timeout
- `src/ida_script_mcp/installer.py`
  - installs plugin support files into IDA plugin directory

V2.3 should **not remove** these pieces. V2.3 should add an isolated execution mode and make it the default for risky script execution.

## 1. V2.3 goal

V2.3 means:

```text
Untrusted / potentially blocking / potentially crashing execute_idapython
  -> always runs in a separate IDA worker process
  -> MCP server can kill that worker process on timeout
  -> current GUI IDA process never executes the dangerous script
  -> GUI IDA only provides context and optionally replays reviewed changes
```

The important design point is that process isolation must not be bypassed by routing the script to the GUI plugin. GUI routing defeats the purpose of V2.3.

## 2. Non-goals

V2.3 should not attempt any of these:

- Do not share one live IDB file between GUI IDA and worker IDA.
- Do not diff raw `.idb` / `.i64` database files as the primary synchronization mechanism.
- Do not execute untrusted user code in the GUI IDA plugin just because GUI has the target database open.
- Do not claim that a native IDA/C API call was killed unless the worker process was actually terminated.
- Do not auto-apply worker changes to GUI by default without an explicit apply request.

## 3. High-level architecture

```text
MCP client
  -> ida_script_mcp.server
       -> GUI plugin context endpoint
       -> IsolatedExecutionManager
            -> creates job directory
            -> copies database or uses configured database source
            -> writes request.json and runner.py
            -> launches idat/idat64/ida64 in autonomous mode
            -> waits for result or timeout
            -> kills process tree on hard timeout
            -> reads result.json / changes.json
       -> returns ExecuteResult + optional ChangeSet

GUI IDA plugin
  -> exposes safe context endpoints
  -> never runs untrusted isolated script
  -> can apply reviewed ChangeSet through /apply_changes

Headless IDA worker process
  -> opens copied database / temporary database
  -> runs bootstrap runner
  -> installs change recorder
  -> executes user script
  -> emits structured result and structured changes
  -> exits normally or is killed by server
```

## 5. Data model

### 5.1 Execute isolation mode

Extend MCP input and shared protocol with an explicit isolation mode:

```python
from typing import Literal, Optional

IsolationMode = Literal["auto", "isolated", "in_process"]

class ExecuteScriptInput(InstanceTargetInput):
    code: Optional[str] = None
    script_path: Optional[str] = None
    capture_output: StrictBool = True
    timeout_seconds: int = Field(default=30, ge=1, le=600)
    isolation: IsolationMode = "auto"
    collect_changes: StrictBool = True
    apply_changes: StrictBool = False
```

Semantics:

- `isolated`: always run in worker process.
- `auto`: V2.3 default; run in worker process for `execute_idapython` unless an internal trusted call explicitly opts out.
- `in_process`: compatibility escape hatch for trusted/local debugging only. It should be clearly documented as unsafe and should not be selected automatically for arbitrary MCP requests.

### 5.2 Isolated job request

Add `IsolatedExecuteRequest`:

```python
class IsolatedExecuteRequest(BaseModel):
    model_config = ConfigDict(extra="forbid")

    execute: ExecuteRequest
    job_id: str
    database_path: str
    input_file_path: Optional[str] = None
    context: dict[str, Any] = Field(default_factory=dict)
    collect_changes: bool = True
    output_dir: str
```

### 5.3 Isolated result

Extend result with worker fields:

```python
IsolatedStatus = Literal[
    "ok",
    "timeout",
    "script_error",
    "source_error",
    "worker_crashed",
    "worker_start_error",
    "worker_result_missing",
]

class IsolatedExecuteResult(ExecuteResult):
    isolated: bool = True
    job_id: str
    worker_pid: Optional[int] = None
    worker_exit_code: Optional[int] = None
    hard_timeout: bool = False
    killed: bool = False
    changes: list[ChangeOperation] = Field(default_factory=list)
    artifacts: dict[str, str] = Field(default_factory=dict)
```

The existing `ExecuteResult.status` may remain the top-level status, but V2.3 should avoid ambiguous `plugin_response_timeout`. In isolated mode the server owns the child process and can return `timeout` with `killed=true` when it actually kills the worker.

## 6. Change protocol

All worker-to-GUI synchronization should use structured operations, not IDB file sharing.

Create `change_protocol.py` with a discriminated union like:

```python
class ChangeBase(BaseModel):
    op_id: str
    op: str
    ea: Optional[int] = None
    source: Literal["explicit_api", "monkeypatch", "snapshot_diff"]
    confidence: Literal["high", "medium", "low"] = "high"
    reason: Optional[str] = None

class RenameChange(ChangeBase):
    op: Literal["rename"] = "rename"
    ea: int
    old_name: Optional[str] = None
    new_name: str
    flags: int = 0

class CommentChange(ChangeBase):
    op: Literal["comment"] = "comment"
    ea: int
    text: str
    repeatable: bool = False

class FunctionCommentChange(ChangeBase):
    op: Literal["function_comment"] = "function_comment"
    ea: int
    text: str
    repeatable: bool = False

class PatchBytesChange(ChangeBase):
    op: Literal["patch_bytes"] = "patch_bytes"
    ea: int
    old_bytes_hex: Optional[str] = None
    new_bytes_hex: str

class TypeChange(ChangeBase):
    op: Literal["set_type"] = "set_type"
    ea: int
    decl: str
    flags: int = 0

class EnumChange(ChangeBase):
    op: Literal["enum"] = "enum"
    name: str
    members: list[dict[str, Any]] = Field(default_factory=list)

class StructChange(ChangeBase):
    op: Literal["struct"] = "struct"
    name: str
    members: list[dict[str, Any]] = Field(default_factory=list)

ChangeOperation = Annotated[
    RenameChange
    | CommentChange
    | FunctionCommentChange
    | PatchBytesChange
    | TypeChange
    | EnumChange
    | StructChange,
    Field(discriminator="op"),
]

class ChangeSet(BaseModel):
    model_config = ConfigDict(extra="forbid")

    schema_version: Literal[1] = 1
    job_id: str
    database_fingerprint: dict[str, Any]
    operations: list[ChangeOperation] = Field(default_factory=list)
    warnings: list[str] = Field(default_factory=list)
```

Use integer addresses internally and JSON serialization mode for output. If the UI/user prefers hex, render hex in presentation, not as the canonical internal type.

## 7. Change capture strategy

V2.3 means automatic capture of common IDAPython write APIs. Implement in layers.

### 7.1 Explicit safe API first

Expose `mcp_changes` into the worker script namespace:

```python
mcp_changes.rename(ea, name, flags=0)
mcp_changes.comment(ea, text, repeatable=False)
mcp_changes.function_comment(ea, text, repeatable=False)
mcp_changes.patch_bytes(ea, data)
mcp_changes.set_type(ea, decl, flags=0)
mcp_changes.struct(name, members)
mcp_changes.enum(name, members)
```

These functions should both:

1. Apply the change inside the worker database, so the script can observe its own changes.
2. Record a structured `ChangeOperation` for GUI replay.

### 7.2 Monkeypatch common write APIs

Patch common write APIs during worker execution so existing scripts produce change logs:

- `idc.set_name`
- `ida_name.set_name`
- `idc.set_cmt`
- `ida_bytes.set_cmt`
- `idc.set_func_cmt`
- `ida_funcs.set_func_cmt`
- `idc.patch_byte`
- `idc.patch_word`
- `idc.patch_dword`
- `idc.patch_qword`
- `ida_bytes.patch_byte`
- `ida_bytes.patch_word`
- `ida_bytes.patch_dword`
- `ida_bytes.patch_qword`
- `ida_bytes.patch_bytes`
- `idc.SetType` if available
- `idc.set_type` if available
- `ida_typeinf.apply_tinfo` where feasible

Monkeypatch wrapper rules:

- Preserve original function behavior and return value.
- Record only if the original call succeeded.
- Capture old values before the call when cheap and safe.
- On wrapper failure, add warning but do not crash the user script unless the original API crashed.
- Deduplicate repeated identical operations if safe.

### 7.3 Optional snapshot diff later

Snapshot diff is lower confidence and should not be required for first V2.3 merge. It can compare selected metadata before/after execution:

- names over functions and named addresses
- comments over touched addresses if tracked
- patched bytes over patched ranges if tracked

Do not perform a whole-database expensive diff by default.

## 8. GUI apply_changes endpoint

Add a GUI plugin endpoint:

```text
POST /apply_changes
```

Request:

```json
{
  "schema_version": 1,
  "job_id": "...",
  "database_fingerprint": {...},
  "operations": [...],
  "dry_run": true
}
```

Response:

```json
{
  "status": "ok" | "partial" | "rejected" | "error",
  "job_id": "...",
  "dry_run": true,
  "applied": [],
  "skipped": [],
  "errors": []
}
```

Rules:

- Default should be `dry_run=true` unless MCP tool explicitly calls an apply tool.
- Check database fingerprint before applying.
- Apply on IDA main thread with write mode.
- Stop on unsafe mismatch by default.
- Return per-operation results.
- Never apply changes from a worker to a different database silently.

Recommended MCP tools:

```text
execute_idapython(..., isolation="isolated", collect_changes=true)
preview_worker_changes(job_id or changes)
apply_worker_changes(changes, dry_run=true|false)
```

## 9. Database fingerprint

Add a shared helper that both GUI and worker can produce:

```python
class DatabaseFingerprint(BaseModel):
    input_file_path: Optional[str]
    database_path: Optional[str]
    root_filename: Optional[str]
    imagebase: Optional[int]
    input_md5: Optional[str]
    input_sha256: Optional[str]
    processor: Optional[str]
    bitness: Optional[int]
```

A strict match should require at least one strong identity:

- same input SHA-256, or
- same input MD5 and root filename, or
- same database path when intentionally operating on a copied IDB lineage

For GUI replay, prefer input hash + imagebase + processor/bitness checks. Paths alone are not sufficient.

## 10. Worker process launch

### 10.1 Configuration

Add environment/config support:

```text
IDA_SCRIPT_MCP_IDA_PATH        path to ida/idat/idat64/ida64 executable
IDA_SCRIPT_MCP_WORK_DIR        base directory for isolated jobs
IDA_SCRIPT_MCP_KEEP_JOBS       0/1, keep job dirs for debugging
IDA_SCRIPT_MCP_WORKER_MODE     idat | ida | auto
```

`auto` should discover likely executables but must also produce clear errors if not found.

### 10.2 Job directory layout

```text
<work_dir>/<job_id>/
  request.json
  runner.py
  user_code.py            optional, when code input is used
  input.idb/.i64          optional copied database
  stdout.txt
  stderr.txt
  result.json
  changes.json
  worker.log
```

### 10.3 Launch command shape

For IDA command-line execution, implement a command builder. The exact executable differs by platform and user install.

Expected shape:

```text
<ida_exe> -A -S<runner.py> <database_path>
```

PowerShell example for a reviewer with IDA installed:

```powershell
$env:IDA_SCRIPT_MCP_IDA_PATH = 'C:\Path\To\IDA\idat64.exe'
$env:IDA_SCRIPT_MCP_WORK_DIR = "$PWD\.ida-script-mcp-jobs"
python -m pytest -q tests/test_isolated_manager.py
```

Important implementation detail: do not build one shell string. Use `subprocess.Popen(args: list[str], ...)` with separate args.

### 10.4 Hard timeout and process-tree kill

`IsolatedExecutionManager` should:

1. Start worker with stdout/stderr redirected to files.
2. Wait for `timeout_seconds + hard_timeout_margin_seconds`.
3. If still running, kill the entire process tree.
4. Return `status="timeout"`, `hard_timeout=true`, `killed=true`.

PowerShell 7 CI/unit tests should mock `subprocess.Popen`; do not require real IDA for normal tests.

Implement process tree kill cross-platform:

- Windows: `taskkill /PID <pid> /T /F` as a fallback.
- POSIX: start new session/process group, kill process group.
- Unit-test command construction and timeout paths with fake processes.

## 11. Worker runner responsibilities

`worker_runner.py` should be able to run inside IDA Python. It should:

1. Locate `request.json` from an environment variable or runner argument.
2. Load `IsolatedExecuteRequest`.
3. Set IDA batch/autonomous-friendly behavior where available.
4. Wait for auto-analysis if requested/configured.
5. Build execution namespace using the existing `_build_ida_globals` concept.
6. Install `ChangeRecorder`.
7. Execute `ExecuteRequest` using `ScriptExecutor` or a worker-compatible equivalent.
8. Write `result.json` atomically.
9. Write `changes.json` atomically.
10. Exit IDA with a deterministic code when possible.

Atomic write pattern:

```python
with open(temp_path, "w", encoding="utf-8") as handle:
    json.dump(payload, handle, ensure_ascii=False, indent=2)
os.replace(temp_path, final_path)
```

Do not depend on the GUI HTTP plugin being loaded inside the worker.

## 12. Server-side routing rules

Current `execute_idapython` should change to:

```text
if isolation in ("auto", "isolated"):
    run IsolatedExecutionManager
else:
    run existing in-process/plugin path, but mark unsafe/legacy
```

Recommended default:

- `auto` -> isolated worker for arbitrary MCP `execute_idapython`.
- Keep GUI `/execute` endpoint for compatibility and trusted debugging, but do not use it for V2.3 default.

Read-only dedicated tools (`list_functions`, `decompile_function`, `get_xrefs`) may continue to use the GUI plugin because they are structured and bounded. The isolation requirement is specifically for arbitrary script execution.

Expected:

- result status `timeout`
- `hard_timeout=true`

### 14.3 Native/blocking behavior

Execute a script that blocks inside an IDA/native or sleep call longer than timeout.

- worker process killed by server

## 15. Security and safety notes

- This is not a sandbox. A worker process can still access files/network according to OS permissions. V2.3 only protects the GUI IDA process from hangs/crashes and provides hard timeout via process kill.
- Do not pass secrets in environment variables to worker unless necessary.
- Job directories may contain user code, stdout/stderr, and analysis output. Respect `IDA_SCRIPT_MCP_KEEP_JOBS` and clean up by default.
- Do not auto-apply changes generated by arbitrary scripts.
- Validate database fingerprint before applying changes.
- Keep localhost-only behavior for GUI plugin HTTP endpoints.

## 17. Acceptance criteria

V2.3 is complete when all are true:

- `execute_idapython` default path starts an external worker process for arbitrary scripts.
- Server kills worker process on hard timeout and returns `killed=true`.
- GUI IDA remains responsive when worker times out or crashes.
- Worker emits structured results and structured changes.
- GUI changes are not auto-applied; they require preview/apply flow.
- Apply endpoint validates database fingerprint and reports per-operation status.
- Unit tests pass without IDA installed.
- Manual IDA 8.3 validation checklist has been run and documented by a maintainer.

## 19. Open design questions for reviewer

1. Should isolated mode copy the existing `.i64/.idb` database, open the original read-only, or rebuild from input binary by default?
2. How should GUI plugin expose dirty/unsaved state? A worker based on the saved database will not see unsaved GUI changes.
3. Which write APIs should be mandatory for V2.3 monkeypatch capture, and which can be deferred?
4. Should `apply_worker_changes` require a separate explicit MCP tool call, or can `execute_idapython(apply_changes=true)` be allowed for trusted users?
5. Should job directories be retained on failure by default for debugging, or always cleaned unless `IDA_SCRIPT_MCP_KEEP_JOBS=1`?

Recommended answers for first implementation:

- Copy saved database by default; warn if GUI reports dirty/unsaved state.
- Require explicit `apply_worker_changes`; keep `execute_idapython(apply_changes=true)` disabled until later.
- Implement explicit `mcp_changes` API before monkeypatch capture.
- Retain failed job directories by default only when debug env var is set.
