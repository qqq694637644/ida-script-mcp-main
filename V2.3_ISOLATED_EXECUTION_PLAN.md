# V2.3 Breaking Isolated Execution + Strict GUI Change Replay Plan

This document is written for another AI/code maintainer to review and implement. The expected shell is **PowerShell 7 (`pwsh`)** from the repository root.

This revision is intentionally a **breaking V2.3 plan**. It removes public compatibility escape hatches so implementation bugs are exposed during development instead of being hidden by fallback to the current GUI `/execute` path.

## 0. Hard position

V2.3 is not a small compatible extension. V2.3 changes the public execution contract:

```text
Public execute_idapython
  -> isolated worker process only
  -> no public in_process mode
  -> no auto fallback to GUI /execute
  -> no execute+apply in one call
  -> worker failure is returned as failure, never hidden
```

The existing V1.5 code on this branch is useful as implementation material, but V2.3 must be allowed to remove or reject unsafe public paths.

## 1. Current branch context

Current branch already contains a V1.5-style refactor:

- `src/ida_script_mcp/protocol.py`
  - strict `ExecuteRequest` / `ExecuteResult` / `ExecutionError` models
  - currently includes `plugin_response_timeout`, which must be removed from the isolated public path
- `src/ida_script_mcp/execution.py`
  - in-process `ScriptExecutor`
  - Python bytecode soft timeout via trace hook
- `src/ida_script_mcp/ida_plugin.py`
  - currently exposes `/execute`, which must no longer be a public arbitrary script endpoint in V2.3
  - `/health` exposes execution state and can stay
- `src/ida_script_mcp/server.py`
  - currently can forward arbitrary code to GUI `/execute`; V2.3 must stop doing that
- `src/ida_script_mcp/installer.py`
  - installs plugin support files into IDA plugin directory

V2.3 may reuse strict model code and executor code inside the worker, but the public MCP path must be isolated-only.

## 2. V2.3 goal

V2.3 means:

```text
Untrusted / potentially blocking / potentially crashing execute_idapython
  -> always runs in a separate IDA worker process
  -> MCP server can kill that worker process on timeout
  -> current GUI IDA process never executes the dangerous script
  -> GUI IDA only provides context and optionally replays reviewed changes
```

Process isolation must not be bypassed. If the worker cannot start, the database cannot be copied, IDA is not configured, or GUI state is dirty, return an explicit failure. Do not fallback to GUI execution.

## 3. Non-goals and forbidden fallbacks

V2.3 must not attempt or allow any of these:

- Do not share one live IDB file between GUI IDA and worker IDA.
- Do not diff raw `.idb` / `.i64` database files as the primary synchronization mechanism.
- Do not execute untrusted user code in the GUI IDA plugin because GUI has the target database open.
- Do not claim that a native IDA/C API call was killed unless the worker process was actually terminated.
- Do not auto-apply worker changes to GUI from `execute_idapython`.
- Do not expose public `isolation="in_process"`.
- Do not expose public `isolation="auto"` if it could fallback to GUI execution.
- Do not keep MCP-server fallback to GUI `/execute` for arbitrary scripts.
- Do not silently downgrade recorder/monkeypatch failures to warnings in development mode.

## 4. High-level architecture

```text
MCP client
  -> ida_script_mcp.server
       -> GUI plugin context endpoint, safe only
       -> IsolatedExecutionManager
            -> creates job directory
            -> requires GUI database to be saved and clean
            -> copies the saved GUI database to the job directory
            -> writes request.json and runner.py
            -> launches idat/idat64/ida64 in autonomous mode
            -> waits for result or hard timeout
            -> kills worker process tree on hard timeout
            -> reads result.json / changes.json
       -> returns ExecuteResult + optional ChangeSet

GUI IDA plugin
  -> exposes safe context endpoints
  -> rejects public arbitrary /execute by default
  -> can apply reviewed ChangeSet through /apply_changes

Headless IDA worker process
  -> opens copied database only
  -> runs bootstrap runner
  -> installs strict change recorder
  -> executes user script
  -> emits structured result and structured changes
  -> exits normally or is killed by server
```

## 5. Public API contract

### 5.1 Public execute input

There is no public `IsolationMode` enum in V2.3. The public MCP tool is isolated-only.

```python
class ExecuteScriptInput(InstanceTargetInput):
    model_config = ConfigDict(extra="forbid", strict=True)

    code: Optional[str] = None
    script_path: Optional[str] = None
    capture_output: StrictBool = True
    timeout_seconds: int = Field(default=30, ge=1, le=600)
    collect_changes: StrictBool = True
```

Hard rules:

- `code` XOR `script_path` exactly one.
- No `isolation` parameter.
- No `apply_changes` parameter.
- No `in_process` parameter.
- Unknown fields are rejected.
- `execute_idapython` always creates an isolated job.
- Worker startup/configuration problems return explicit failure statuses.

### 5.2 Dev-only unsafe GUI execute

If maintainers need a debugging entrypoint, it must be non-public and disabled by default:

```text
IDA_SCRIPT_MCP_ENABLE_UNSAFE_GUI_EXECUTE=1
```

Behavior:

- Default GUI `/execute` response must be `410 Gone` or `501 Not Implemented` for arbitrary script execution.
- Response body must say to use isolated worker execution.
- Tests must assert GUI `/execute` is rejected by default.
- If env-gated dev-only behavior exists, tests must assert it is unavailable unless the env var is explicitly set.
- MCP server must never use this dev-only endpoint for public `execute_idapython`.

## 6. Status model

Update `ExecuteStatus` so isolated worker statuses are valid at the shared model layer:

```python
ExecuteStatus = Literal[
    "ok",
    "timeout",
    "script_error",
    "source_error",
    "worker_crashed",
    "worker_start_error",
    "worker_result_missing",
    "recorder_error",
    "rejected",
]
```

Remove `plugin_response_timeout` from the isolated public path. It is only meaningful for the old plugin HTTP path, which V2.3 must not use for arbitrary public execution.

Status semantics:

- `ok`: worker executed and produced a valid result.
- `timeout`: server killed the worker process tree after hard timeout; result must include `killed=true`.
- `script_error`: user script raised after worker and recorder were correctly initialized.
- `source_error`: invalid request source, missing script file, invalid code, missing clean database source, or database copy source problem.
- `worker_crashed`: worker exited non-zero without a valid structured script error/result.
- `worker_start_error`: IDA executable missing, command construction failed, process start failed, recorder install failed before user script, or required worker bootstrap failed.
- `worker_result_missing`: worker exited zero or non-zero but expected result files are missing or unreadable.
- `recorder_error`: change recorder wrapper failed during execution or failed to serialize changes.
- `rejected`: policy rejection, such as GUI dirty/unsaved state or attempted unsupported apply-in-execute behavior.

Do not subclass `ExecuteResult` with a status type that contains values missing from `ExecuteResult.status`. Either extend `ExecuteStatus` directly or define a separate model that does not inherit incompatible status fields.

Recommended result model:

```python
class ExecuteResult(BaseModel):
    model_config = ConfigDict(extra="forbid")

    status: ExecuteStatus
    result: Any = None
    stdout: str = ""
    stderr: str = ""
    error: Optional[ExecutionError] = None
    duration_seconds: float = Field(default=0.0, ge=0.0)
    timeout_seconds: int = Field(default=30, ge=1)
    instance_id: Optional[str] = None
    port: Optional[int] = None
    isolated: bool = True
    job_id: Optional[str] = None
    worker_pid: Optional[int] = None
    worker_exit_code: Optional[int] = None
    killed: bool = False
    hard_timeout: bool = False
    changes: list[ChangeOperation] = Field(default_factory=list)
    artifacts: dict[str, str] = Field(default_factory=dict)
```

## 7. Isolated job request

Add `IsolatedExecuteRequest`:

```python
class IsolatedExecuteRequest(BaseModel):
    model_config = ConfigDict(extra="forbid")

    execute: ExecuteRequest
    job_id: str
    database_path: str
    database_copy_path: str
    input_file_path: Optional[str] = None
    context: dict[str, Any] = Field(default_factory=dict)
    collect_changes: bool = True
    output_dir: str
```

Rules:

- `database_path` is the saved GUI database path used as source.
- `database_copy_path` is the copied database inside the job directory that the worker opens.
- Worker must never open the live GUI database for arbitrary script execution.

## 8. Database source policy: fixed first implementation

This is not an open design question in V2.3 first implementation.

First implementation policy:

```text
worker database source = copy of GUI current saved database
```

Hard rules:

- GUI plugin must expose `database_path`, `database_fingerprint`, and `dirty`/`unsaved` state if possible.
- If GUI reports dirty/unsaved, `execute_idapython` returns `rejected`.
- Do not auto-save GUI database.
- Do not fallback to GUI `/execute`.
- Do not rebuild from input binary in first implementation.
- Do not open original GUI database read-only in first implementation.
- If `database_path` is missing, unreadable, or copy fails, return `source_error` or `worker_start_error` depending on where the failure occurs.

Suggested classification:

- no GUI target / no database path: `source_error`
- GUI dirty/unsaved: `rejected`
- copy source missing/unreadable: `source_error`
- copy destination/write failure: `worker_start_error`

## 9. Database fingerprint

Add a shared helper that both GUI and worker can produce:

```python
class DatabaseFingerprint(BaseModel):
    model_config = ConfigDict(extra="forbid")

    input_file_path: Optional[str]
    database_path: Optional[str]
    root_filename: Optional[str]
    imagebase: Optional[int]
    input_md5: Optional[str]
    input_sha256: Optional[str]
    processor: Optional[str]
    bitness: Optional[int]
```

Strict replay requires a strong identity match:

- same input SHA-256, or
- same input MD5 plus same root filename and imagebase, or
- explicit same copied-database lineage recorded in the job metadata.

Paths alone are not sufficient.

## 10. Change protocol

All worker-to-GUI synchronization uses structured operations, not IDB file sharing.

Create `change_protocol.py` with a discriminated union like:

```python
class ChangeBase(BaseModel):
    model_config = ConfigDict(extra="forbid")

    op_id: str
    op: str
    ea: Optional[int] = None
    source: Literal["explicit_api", "monkeypatch"]
    confidence: Literal["high", "medium", "low"] = "high"
    reason: Optional[str] = None

class RenameChange(ChangeBase):
    op: Literal["rename"] = "rename"
    ea: int
    old_name: Optional[str] = None
    new_name: str
    flags: int = 0

class CommentChange(ChangeBase):
    op: Literal["comment"] = "comment"
    ea: int
    text: str
    repeatable: bool = False

class FunctionCommentChange(ChangeBase):
    op: Literal["function_comment"] = "function_comment"
    ea: int
    text: str
    repeatable: bool = False

class PatchBytesChange(ChangeBase):
    op: Literal["patch_bytes"] = "patch_bytes"
    ea: int
    old_bytes_hex: Optional[str] = None
    new_bytes_hex: str

class TypeChange(ChangeBase):
    op: Literal["set_type"] = "set_type"
    ea: int
    decl: str
    flags: int = 0

ChangeOperation = Annotated[
    RenameChange
    | CommentChange
    | FunctionCommentChange
    | PatchBytesChange
    | TypeChange,
    Field(discriminator="op"),
]

class ChangeSet(BaseModel):
    model_config = ConfigDict(extra="forbid")

    schema_version: Literal[1] = 1
    job_id: str
    database_fingerprint: DatabaseFingerprint
    operations: list[ChangeOperation] = Field(default_factory=list)
```

Do not include `warnings` as a recorder-failure escape hatch. Recorder failures should produce `recorder_error` or `worker_start_error`.

## 11. Strict change capture strategy

V2.3 automatic change capture is strict. A write that cannot be recorded is a failure because otherwise the GUI replay log can silently diverge from worker behavior.

### 11.1 Explicit safe API

Expose `mcp_changes` into the worker script namespace:

```python
mcp_changes.rename(ea, name, flags=0)
mcp_changes.comment(ea, text, repeatable=False)
mcp_changes.function_comment(ea, text, repeatable=False)
mcp_changes.patch_bytes(ea, data)
mcp_changes.set_type(ea, decl, flags=0)
```

These functions must both:

1. Apply the change inside the worker database, so the script can observe its own changes.
2. Record a structured `ChangeOperation` for GUI replay.

If applying succeeds but recording fails, worker must return `recorder_error`. Do not continue as warning-only.

### 11.2 Monkeypatch common write APIs

Patch common write APIs during worker execution so existing scripts produce change logs:

- `idc.set_name`
- `ida_name.set_name`
- `idc.set_cmt`
- `ida_bytes.set_cmt`
- `idc.set_func_cmt`
- `ida_funcs.set_func_cmt`
- `idc.patch_byte`
- `idc.patch_word`
- `idc.patch_dword`
- `idc.patch_qword`
- `ida_bytes.patch_byte`
- `ida_bytes.patch_word`
- `ida_bytes.patch_dword`
- `ida_bytes.patch_qword`
- `ida_bytes.patch_bytes`
- `idc.SetType` if available
- `idc.set_type` if available
- `ida_typeinf.apply_tinfo` where feasible

Strict wrapper rules:

- Recorder install/monkeypatch install failure -> `worker_start_error` before user script starts.
- Wrapper calls original function and preserves original return value only if recording succeeds.
- Record only if the original call succeeded according to that API's return semantics.
- Capture old values before the call when required for replay safety.
- Wrapper recording failure -> `recorder_error`, fail the worker.
- Original IDA API exception -> preserve as `script_error`.
- Original IDA API false/failed return -> preserve original return and record a rejected/failed operation only if the API semantics are known; otherwise do not invent success.
- Deduplication must not drop semantically distinct operations.

### 11.3 Snapshot diff deferred

Snapshot diff is not part of first V2.3 acceptance. It can be added later as an explicit lower-confidence mode, but not as a fallback for failed recorder monkeypatches.

## 12. GUI apply_changes endpoint

Add a GUI plugin endpoint:

```text
POST /apply_changes
```

Request:

```json
{
  "schema_version": 1,
  "job_id": "...",
  "database_fingerprint": {...},
  "operations": [...],
  "dry_run": true
}
```

Response:

```json
{
  "status": "ok" | "partial" | "rejected" | "error",
  "job_id": "...",
  "dry_run": true,
  "applied": [],
  "skipped": [],
  "errors": []
}
```

Rules:

- `execute_idapython` never applies changes.
- Public `execute_idapython` does not accept `apply_changes`.
- Applying changes requires a separate `apply_worker_changes` MCP tool call.
- `apply_worker_changes` defaults to `dry_run=true`.
- `dry_run=false` must be explicitly provided.
- Check database fingerprint before applying.
- Apply on IDA main thread with write mode.
- Stop on unsafe mismatch by default.
- Return per-operation results.
- Never apply changes from a worker to a different database silently.

Recommended MCP tools:

```text
execute_idapython(code|script_path, timeout_seconds, collect_changes)
preview_worker_changes(job_id or changes)
apply_worker_changes(changes, dry_run=true|false)
```

`preview_worker_changes` can be a convenience wrapper over `apply_worker_changes(..., dry_run=true)`.

## 13. Worker process launch

### 13.1 Configuration

Add environment/config support:

```text
IDA_SCRIPT_MCP_IDA_PATH        path to ida/idat/idat64/ida64 executable
IDA_SCRIPT_MCP_WORK_DIR        base directory for isolated jobs
IDA_SCRIPT_MCP_KEEP_JOBS       0/1, keep job dirs for debugging
IDA_SCRIPT_MCP_WORKER_MODE     idat | ida | auto
```

`IDA_SCRIPT_MCP_WORKER_MODE=auto` may discover likely executables, but discovery failure must return `worker_start_error`. It must not fallback to GUI execution.

### 13.2 Job directory layout

```text
<work_dir>/<job_id>/
  request.json
  runner.py
  user_code.py
  database.i64 or database.idb
  stdout.txt
  stderr.txt
  result.json
  changes.json
  worker.log
  metadata.json
```

### 13.3 Launch command shape

Expected shape:

```text
<ida_exe> -A -S<runner.py> <database_copy_path>
```

PowerShell example for a reviewer with IDA installed:

```powershell
$env:IDA_SCRIPT_MCP_IDA_PATH = 'C:\Path\To\IDA\idat64.exe'
$env:IDA_SCRIPT_MCP_WORK_DIR = "$PWD\.ida-script-mcp-jobs"
python -m pytest -q tests/test_isolated_manager.py
```

Implementation detail: do not build one shell string. Use `subprocess.Popen(args: list[str], ...)` with separate args.

### 13.4 Hard timeout and process-tree kill

`IsolatedExecutionManager` must:

1. Start worker with stdout/stderr redirected to files.
2. Wait for `timeout_seconds + hard_timeout_margin_seconds`.
3. If still running, kill the entire process tree.
4. Verify the worker PID/process tree is gone when possible.
5. Return `status="timeout"`, `hard_timeout=true`, `killed=true`.

Process tree kill:

- Windows: use `taskkill /PID <pid> /T /F` as fallback.
- POSIX: start new session/process group, kill process group.
- Unit tests must mock process behavior and assert kill path is invoked.

## 14. Worker runner responsibilities

`worker_runner.py` should be able to run inside IDA Python. It should:

1. Locate `request.json` from an environment variable or runner argument.
2. Load `IsolatedExecuteRequest`.
3. Open the copied database path only.
4. Set IDA batch/autonomous-friendly behavior where available.
5. Wait for auto-analysis if requested/configured.
6. Build execution namespace using the existing `_build_ida_globals` concept.
7. Install `ChangeRecorder` in strict mode before user script starts.
8. Execute `ExecuteRequest` using `ScriptExecutor` or worker-compatible equivalent.
9. Write `result.json` atomically.
10. Write `changes.json` atomically.
11. Exit IDA with a deterministic code when possible.

Atomic write pattern:

```python
with open(temp_path, "w", encoding="utf-8") as handle:
    json.dump(payload, handle, ensure_ascii=False, indent=2)
os.replace(temp_path, final_path)
```

Do not depend on the GUI HTTP plugin being loaded inside the worker.

## 15. Server-side routing rules

Public `execute_idapython` should be:

```text
execute_idapython
  -> resolve target GUI context only
  -> require clean saved database path
  -> copy database to job dir
  -> run IsolatedExecutionManager
  -> return result/changes
```

Forbidden routing:

```text
execute_idapython
  -> GUI /execute
```

Read-only dedicated tools (`list_functions`, `decompile_function`, `get_xrefs`) may continue to use the GUI plugin because they are structured and bounded. The isolation requirement is for arbitrary script execution.

## 18. Security and safety notes

- This is not a sandbox. A worker process can still access files/network according to OS permissions. V2.3 protects the GUI IDA process from hangs/crashes and provides hard timeout via process kill.
- Do not pass secrets in environment variables to worker unless necessary.
- Job directories may contain user code, stdout/stderr, and analysis output. Respect `IDA_SCRIPT_MCP_KEEP_JOBS` and clean up by default.
- Do not auto-apply changes generated by arbitrary scripts.
- Validate database fingerprint before applying changes.
- Keep localhost-only behavior for GUI plugin HTTP endpoints.
- Treat recorder failure as data-integrity failure, not as a warning.

## 19. Suggested implementation phases

### Phase A: breaking public protocol

- Remove public `isolation` and `apply_changes` fields.
- Extend `ExecuteStatus` with worker statuses.
- Remove isolated-path use of `plugin_response_timeout`.
- Make GUI `/execute` reject arbitrary scripts by default.
- Add tests proving MCP public execute does not call GUI `/execute`.

### Phase B: protocols and process manager

- Add `change_protocol.py`.
- Add `isolated_protocol.py`.
- Add `process_utils.py`.
- Add `IsolatedExecutionManager` with mocked tests.
- No real IDA required.

### Phase C: worker runner skeleton

- Add `worker_runner.py`.
- Generate job directory and request files.
- Copy the saved clean GUI database to job directory.
- Run worker with fake executable in tests.
- Read `result.json` and classify worker errors.

### Phase D: real IDA worker support

- Build command for `idat64/idat/ida64`.
- Add config/env discovery.
- Add manual validation instructions.
- Keep tests mocked.

### Phase E: explicit change API

- Add `mcp_changes` safe API.
- Record and emit `changes.json`.
- Add GUI `/apply_changes` dry-run endpoint.
- Add MCP `apply_worker_changes` tool.

### Phase F: strict monkeypatch capture

- Add `ChangeRecorder.install(strict=True)` and `ChangeRecorder.uninstall()`.
- Patch common IDAPython write APIs.
- Recorder install failure -> `worker_start_error`.
- Wrapper recording failure -> `recorder_error`.
- Add unit tests with fake modules.
- Mark monkeypatch-generated operations with `source="monkeypatch"`.

### Phase G: isolated-only public execution

- Public `execute_idapython` always uses isolated worker.
- Worker start/copy/config/dirty errors fail explicitly.
- No fallback to GUI `/execute`.
- Update README and changelog with breaking changes.

## 20. Acceptance criteria

V2.3 is complete when all are true:

- Public `execute_idapython` starts an external worker process for arbitrary scripts.
- Public MCP schema does not contain `isolation="in_process"`.
- Public MCP schema does not contain `apply_changes`.
- Public MCP schema does not contain an `auto` mode that can fallback to GUI execution.
- MCP server never sends arbitrary public scripts to GUI `/execute`.
- GUI `/execute` is rejected/removed by default and tests cover that behavior.
- Worker startup failure returns `worker_start_error` and does not fallback.
- Missing IDA path returns `worker_start_error` and does not fallback.
- Database path missing/copy failure returns explicit failure and does not fallback.
- GUI dirty/unsaved returns `rejected` and does not auto-save or fallback.
- Isolated timeout kills the worker process tree and result has `killed=true`.
- Timeout validation checks worker PID/process tree no longer exists when possible.
- Recorder install failure makes worker fail.

## 22. Fixed decisions for first implementation

These are intentionally not open questions for V2.3 first implementation:

1. Worker uses a copy of the current clean saved GUI database.
2. GUI dirty/unsaved state rejects execution.
3. No auto-save.
4. No fallback to GUI `/execute`.
5. No public `in_process` mode.
6. No public `apply_changes` in `execute_idapython`.
7. `apply_worker_changes` is a separate tool and defaults to dry-run.
8. Recorder/monkeypatch failures are hard failures in strict V2.3.
9. Snapshot diff is deferred and cannot mask recorder failure.

Remaining review questions may refine details such as exact list of monkeypatched APIs, job cleanup policy, and IDA executable discovery, but they must not reintroduce fallback execution in GUI IDA.
